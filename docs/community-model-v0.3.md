# Community Data Model — Design Specification

**Status:** Draft v0.3 · April 2026
**Purpose:** Refactor the platform from a single-tenant global thread pool to a multi-community architecture where each community owns its own deliberations, members, facilitators, and audit trail.
**Audience:** Platform Development project (Claude Code implementation) + Constitution & Strategy (governance implications)

> **⚠ Provenance & staleness notice.** This document was generated by an instance of Claude Opus operating in the Constitution & Strategy project, which does NOT have access to the live codebase. The data model, route structure, and field names described here are based on architecture documents and the decision log as of early April 2026, not on direct inspection of the deployed code. Where this specification contradicts the actual codebase — different field names, evolved relationships, models that have changed since the architecture doc was last updated, additional constraints not documented here — **Claude Code should defer to the live code** and adapt the design to match. Treat this document as a target architecture and migration intent, not as a literal schema. Flag any significant contradictions back to the strategy project so the source documents can be updated.

---

## 1. Why This Change

The current data model treats the platform as a single global space with a flat taxonomy of policy domains (Healthcare, Education, Defense, etc.). All threads are visible to all users; all tiers are global; all facilitators can act on any thread.

This model breaks the moment the platform is deployed for a real community. Redlands residents deliberating on e-bike regulations and Maine residents deliberating on PBM reform share no context, no verification, no facilitators, and no political geography. They should not share a thread pool, a member list, or a trust boundary.

The refactor introduces **Community** as the primary organizational unit. A community is a bounded group of people who share a stake in a set of decisions. Everything that currently hangs off the global namespace — threads, tiers, facilitator appointments, audit logs, funding pools — becomes community-scoped.

### What this enables

- **Local-first deployment.** The platform launches for Redlands as one community, then adds others without architectural changes.
- **Scoped verification.** "Verified Redlands resident" is a different claim from "verified Maine voter." Each community defines its own verification standard.
- **Scoped facilitation.** A Redlands facilitator cannot advance phases on a Maine thread. Facilitator trust is local to the community that granted it.
- **Community as the unit of credibility.** A community that completes five clean deliberation cycles builds a visible track record. The community page *is* the credibility artifact.
- **Multi-tenant GTM.** The platform can be deployed for an HOA, a union local, a city, a state constituency, or a co-op — same infrastructure, different communities, different verification models.

### What this does NOT change

- The phase-gated deliberation model (OPEN → ARCHIVED) is unchanged.
- Structured signals (support/concern/need_info/block) are unchanged.
- The append-only audit log architecture is unchanged.
- Vote immutability and all phase gate enforcement are unchanged.
- The LLM integration plan is unchanged (deferred to Phase 5).
- Posts have no upvotes/downvotes. Signals are per-thread, not per-post.

## 2. Data Model

### 2.1 New: Community

```
Community
├── id              UUID, primary key
├── slug            TEXT, unique, URL-safe (e.g., "redlands", "bitcoin-quantum")
├── name            TEXT (e.g., "City of Redlands")
├── description     TEXT (public-facing summary of what this community deliberates on)
├── community_type  ENUM: geographic | organizational | institutional | topical | technical
├── boundary_desc   TEXT (human-readable: "Residents of the City of Redlands, CA")
├── verification_method  TEXT (describes how membership is verified for this community)
├── is_public       BOOLEAN, default TRUE (can non-members browse threads and audit log?)
├── is_invite_only  BOOLEAN, default FALSE (if TRUE, magic-link self-join is disabled)
├── default_phase_durations  JSONB, nullable (suggested durations per phase, e.g.,
│                                {"OPEN": "3d", "DELIBERATING": "14d", "PROPOSING": "5d", "VOTING": "3d"})
├── is_active       BOOLEAN, default TRUE
├── created_at      TIMESTAMPTZ
├── created_by_id   UUID, FK → users (platform admin who created it)
```

**Design notes:**
- `community_type` is informational, not load-bearing. It includes `topical` (e.g., "Philosophy of Mind") and `technical` (e.g., "Bitcoin Quantum Risk") alongside the original three. The platform treats all types the same; the field exists to help users browse and to inform UI hints.
- `boundary_desc` and `verification_method` are human-readable text, not machine-enforced. Verification is a human process at MVP scale. These fields document *what* the verifier checks, not *how* the system enforces it.
- `is_public` defaults to TRUE. Public browsing of deliberations and audit logs is a core legitimacy mechanism.
- `is_invite_only` is for communities where membership should not be self-service even with a magic link — typically small private communities (e.g., a friends-and-admins community for macroeconomics discussion). When TRUE, joining requires an explicit invitation from a community admin.
- `default_phase_durations` is **suggested**, not enforced. Civic communities working against a legislative clock might use short cycles (4-week total). Topical and academic communities have no external clock and may use much longer durations or none at all (facilitator advances when discussion has matured). The state machine itself is unchanged; this field just informs UI defaults and reminders.
- Communities are created by platform admins only. There is no self-service "create a community" flow in MVP. This prevents spam communities and preserves the deliberate-growth property.

### 2.2 New: CommunityMembership

```
CommunityMembership
├── id              UUID, primary key
├── community_id    UUID, FK → communities, NOT NULL
├── user_id         UUID, FK → users, NOT NULL
├── tier            ENUM: registered | participant | facilitator | admin
├── joined_at       TIMESTAMPTZ
├── verified_at     TIMESTAMPTZ, nullable
├── verified_by_id  UUID, FK → users, nullable
├── is_active       BOOLEAN, default TRUE
├── UNIQUE(community_id, user_id)
```

**This is the most important table in the refactor.** Tiers are now per-community, not global.

A user who signs up via magic link gets a `registered` membership in a community when they join it. A community facilitator verifies them to `participant`. A community admin promotes them to `facilitator`. A platform admin (distinct from community admin) creates communities and seeds their first admin.

**Implications:**
- A user can be a `facilitator` in Redlands and a `registered` member in a different community.
- A user can be a member of multiple communities.
- The `FacilitatorRequest` model (existing) becomes community-scoped: "I am requesting facilitator status in *this* community."
- The global `tier` field on the `users` table becomes a **platform-level** tier, used only for platform admin operations (creating communities, managing the global domain list). It no longer gates thread-level actions.

### 2.3 New: PlatformRole (on existing users table)

```
users (existing table)
├── ...existing fields...
├── platform_role   ENUM: user | platform_admin    (NEW, default: user)
```

This replaces the current global `tier` field for platform-level operations only. A `platform_admin` can create communities and seed community admins. Everything else — thread creation, signals, posts, proposals, votes, facilitation — is gated by `CommunityMembership.tier`, not by any global role.

### 2.4 Modified: Thread

```
Thread (existing, modified)
├── ...existing fields...
├── community_id    UUID, FK → communities, NOT NULL   (NEW)
├── domain_slug     TEXT, FK → domains                 (UNCHANGED — remains a categorization tag)
```

A thread belongs to exactly one community. The domain_slug remains as a taxonomy tag ("public-safety", "infrastructure", "healthcare") but is no longer the primary organizational unit.

**Constraint:** Only members of the thread's community can interact with it (signal, post, propose, vote). Non-members can read if `community.is_public = TRUE`.

### 2.5 Modified: FacilitatorRequest

```
FacilitatorRequest (existing, modified)
├── ...existing fields...
├── community_id    UUID, FK → communities, NOT NULL   (NEW)
```

A facilitator request is now scoped to a specific community. The admin approval queue at `/admin` becomes per-community (community admins see requests for their community only). Platform admins see all.

### 2.6 Modified: FundingPool

```
FundingPool (existing, modified)
├── ...existing fields...
├── community_id    UUID, FK → communities, NOT NULL   (NEW)
```

Funding pools are per-community. A Redlands funding pool does not mingle with a Maine funding pool.

### 2.7 Modified: AuditLog

```
AuditLog (existing, modified)
├── ...existing fields...
├── community_id    UUID, FK → communities, nullable   (NEW)
```

The community_id is nullable because some audit events are platform-level (community creation, platform admin actions). All community-scoped actions (thread creation, phase advances, signals, posts, votes, facilitator promotions) carry the community_id.

The public audit endpoint becomes community-scoped:
- `GET /api/v1/communities/{slug}/audit` → audit log for that community
- `GET /api/v1/audit` → platform-level audit events only (community creation, etc.)

### 2.8 Unchanged Models

These models require no schema changes. They inherit community scope through their parent Thread:

- **Post** — belongs to Thread, which belongs to Community
- **Signal** — belongs to Thread, which belongs to Community
- **Proposal** — belongs to Thread, which belongs to Community
- **Vote** — belongs to Proposal, which belongs to Thread, which belongs to Community
- **AllocationDecision** — belongs to Proposal + FundingPool, both community-scoped

### 2.9 Modified: Domain

```
Domain (existing, modified)
├── slug            TEXT
├── community_id    UUID, FK → communities, NOT NULL   (NEW)
├── name            TEXT
├── is_active       BOOLEAN
├── PRIMARY KEY (community_id, slug)
```

Domains become community-scoped. Each community defines its own taxonomy of categorization tags. Redlands might use "Public Safety," "Land Use," "Infrastructure." A philosophy community might use "Free Will," "Consciousness," "Personal Identity." A Bitcoin Quantum Risk community might use "Signature Schemes," "Migration Paths," "Timeline Estimates." There is no global domain list; each community curates what's appropriate for its deliberations.

The 15 existing seeded domains migrate into the `test` community as its initial taxonomy (see migration plan §6).

## 3. URL Structure

Current:
```
/threads                    → all threads (global)
/thread/{id}                → thread detail
/admin                      → global admin page
```

Proposed:
```
/c/{slug}                   → community home page
/c/{slug}/threads           → thread list for this community
/c/{slug}/thread/{id}       → thread detail (unchanged UX, community-scoped auth)
/c/{slug}/new-thread        → create thread in this community
/c/{slug}/audit             → public audit log for this community
/c/{slug}/admin             → community admin page (facilitator requests, member management)
/c/{slug}/members           → public member list (display names + tiers, no PII)
/c/{slug}/join              → join this community (magic link → registered membership)

/admin                      → platform admin (create communities, manage global domains)
/account                    → user account (unchanged, but now shows community memberships)
```

The `/c/` prefix prevents slug collisions with existing routes (`/signin`, `/account`, `/admin`, `/how-it-works`).

## 4. Community Home Page

This is the single most important new page. It's what a stranger sees when they visit `/c/redlands`. It must answer three questions immediately:

1. **What is this community?** Name, boundary, description, member count.
2. **What's being deliberated right now?** Active threads with phase badges and signal distributions.
3. **What has this community decided before?** Closed/archived threads with outcomes.

It is NOT a feed. It is NOT a forum landing page. It is structured like a civic agenda:

```
┌─────────────────────────────────────────────┐
│ CITY OF REDLANDS                            │
│ 127 verified residents · 3 active           │
│ deliberations · Est. April 2026             │
│                                             │
│ "Residents of the City of Redlands, CA      │
│  deliberating on local policy together."    │
├─────────────────────────────────────────────┤
│ ACTIVE DELIBERATIONS                        │
│                                             │
│ ▸ E-Bike Regulations          DELIBERATING  │
│   Public Safety · 14 signals · 23 posts     │
│                                             │
│ ▸ Wildfire Preparedness Plan  OPEN          │
│   Infrastructure · 3 signals · 0 posts      │
├─────────────────────────────────────────────┤
│ COMPLETED                                   │
│                                             │
│ ▸ (none yet)                                │
├─────────────────────────────────────────────┤
│ [Join This Community]  [View Audit Log]     │
└─────────────────────────────────────────────┘
```

The community home page is the platform's primary credibility artifact. It replaces the need for a pitch deck or marketing site. A council member who visits `/c/redlands` sees: how many verified residents are involved, what they're deliberating on, and the full audit trail of every action taken. That page *is* the pitch.

## 5. Authorization Model

### Current (global tiers)

```
Action              Required tier (global)
────────────────    ─────────────────────
Read threads        None
Cast signal         registered
Create thread       registered
Create post         registered (was participant, lowered for MVP)
Submit proposal     registered
Cast vote           registered
Advance phase       facilitator
Approve requests    admin
```

### Proposed (community-scoped tiers)

```
Action                  Required community tier    Notes
──────────────────────  ────────────────────────    ─────────────────────────────
Read threads            None (if community public)  Non-members can browse
Join community          None (creates registered    Magic link required
                        membership)
Cast signal             registered                  In target community
Create thread           registered                  In target community
Create post             participant                 Restore to participant tier
Submit proposal         participant                 Restore to participant tier
Cast vote               participant                 Restore to participant tier
Advance phase           facilitator                 In target community
Approve requests        admin                       Community admin, not platform
Create community        platform_admin              Global role, not community

```

**Note on tier restoration:** The foundational documents lowered thread creation and posting to `registered` for MVP testing convenience. With the community model, verification becomes scoped and more tractable (a Redlands facilitator verifies Redlands residents, not the entire world). This makes it practical to restore `participant` as the gate for substantive actions (posting, proposing, voting) — which is what the original design intended. Thread creation and signaling can stay at `registered` since they are organizational/expressive rather than deliberative.

This is a design recommendation, not a hard requirement. It should be discussed before implementation.

## 6. Migration Path

The current deployed platform has existing data (users, threads, domains, audit logs). The migration must preserve all of it.

### Step 1: Create Community table
Standard Alembic migration. No data dependencies.

### Step 2: Create the `test` community
Insert a single community record to absorb existing legacy data:
```python
Community(
    slug="test",
    name="Test / Legacy",
    description="Pre-community-era threads and seed data. Used for ongoing testing.",
    community_type="institutional",
    boundary_desc="Legacy platform members and ongoing test data",
    verification_method="Email magic link (legacy global tier, no scoped verification)",
    is_public=True,
    is_invite_only=False,
    is_active=True,
)
```

All existing threads, domains, audit logs, and facilitator requests get attached to this community. It is not the platform commons — it is explicitly the legacy/test sandbox. The platform commons is created fresh in step 7.

### Step 3: Add community_id to Thread, FacilitatorRequest, FundingPool, AuditLog
Add as nullable first, backfill with default community ID, then set NOT NULL.

### Step 4: Create CommunityMembership table
Populate from existing users:
```sql
INSERT INTO community_memberships (community_id, user_id, tier, joined_at)
SELECT
    '{default_community_id}',
    id,
    tier,         -- copy current global tier
    created_at
FROM users;
```

### Step 5: Add platform_role to users table
Set all existing admins to `platform_admin`. Set everyone else to `user`.

### Step 6: Update API routes
All routes that currently check global `tier` must check `CommunityMembership.tier` for the relevant community. The community is identified from the thread's `community_id` (for thread-scoped actions) or from the URL path (for community-scoped actions like joining, creating threads, viewing audit log).

This is the largest code change and should be done route-by-route with tests for each.

### Step 7: Create the initial real communities

```python
# The Platform Commons — fresh, clean meta-community for cross-community
# discussion and platform governance
Community(
    slug="civic-power-consortium",
    name="Civic Power Consortium",
    description="Platform-wide discussion of civic deliberation, platform governance, and topics that span communities.",
    community_type="institutional",
    boundary_desc="All platform members interested in the platform itself",
    verification_method="Email magic link",
    is_public=True,
    is_invite_only=False,
)

# The first geographic civic deployment
Community(
    slug="redlands",
    name="City of Redlands",
    description="Verified Redlands residents deliberating on local policy.",
    community_type="geographic",
    boundary_desc="Residents of the City of Redlands, California",
    verification_method="Manual verification by community admin (residency check)",
    is_public=True,
    is_invite_only=False,
    default_phase_durations={"OPEN": "3d", "DELIBERATING": "14d", "PROPOSING": "5d", "VOTING": "3d"},
)

# Private friends-and-admins community for macroeconomics, monetary policy,
# institutions, incentive structures, investing — replaces group chat with
# durable structured discussion
Community(
    slug="macro-circle",
    name="Macro Circle",
    description="Private community for ongoing discussion of macro policy, monetary systems, institutions, and investing.",
    community_type="topical",
    boundary_desc="Invite-only group of friends and admins",
    verification_method="Direct invitation by admin",
    is_public=False,
    is_invite_only=True,
)
```

The Bitcoin Quantum Risk community is documented in §11 as a planned future deployment but is not created in the initial migration. It requires a respected Bitcoin developer as founding admin, which is a separate effort.

### Step 8: Build the community home page

New page at `/c/{slug}` showing community name, member count, active deliberation count, description, list of active threads grouped by phase, list of recently closed threads with outcomes, and links to the community audit log and member list. This page is the primary public-facing artifact of each community — it replaces the global `/threads` view as the entry point a stranger sees. See §4 for the layout intent. The page must work for unauthenticated visitors when `community.is_public` is true.

### Step 9: Update frontend routes and templates to be community-aware

Existing pages that need community context: thread list (scoped to one community), thread detail (auth checks against community membership), new-thread form (community pre-selected from URL), account page (shows list of community memberships), admin page (split into community admin and platform admin views). The `api.js` module must continue to contain only fetch calls — no DOM manipulation — so it survives the eventual React migration. New routes follow the `/c/{slug}/...` pattern documented in §3. Old routes (`/threads`, `/thread/{id}`) redirect to the corresponding `/c/test/...` paths.

## 7. What This Does NOT Solve (Yet)

Decisions that are surfaced by this model but not resolved here:

| Question | Notes |
|---|---|
| **Cross-community identity** | Can a platform verify someone once and have it count in multiple communities? Not needed for MVP (each community verifies independently) but matters at scale. |
| **Community discovery** | How does someone find communities to join? Not needed when there are 1–3 communities. Matters when there are 50. |
| **Community creation governance** | Who can request a new community? What criteria must be met? For MVP: platform admin creates manually. At scale: needs a process. |
| **Inter-community deliberation** | Can two communities co-deliberate on a shared issue (e.g., neighboring cities on a regional transport plan)? Not needed for MVP. |
| **Community forking** | Hirschman's exit right applied to communities. If a community's facilitators are captured, can members fork it? The open-source codebase enables this in principle but the data portability mechanism needs specification. |

## 8. Resolved Decisions

**Tier requirements (RESOLVED 2026-04-11):** All substantive actions remain at `registered` tier for now. Friction reduction is the priority during the local-first MVP phase, since the verification step would otherwise block engagement before the deliberation cycle even begins. The `participant` tier remains in the schema as a reserved future state, to be activated when scoped verification (e.g., voter file cross-reference for Redlands) is built. The community model still gates by community membership — a registered member of Redlands cannot act in another community without joining it — which provides a weaker but still meaningful trust boundary.

**Community admin succession (RESOLVED 2026-04-11):** The builder is the first community admin for Redlands and will personally seed the first admin of each early community. Community admin governance, succession rules, and term limits are deferred to the governance charter and will be addressed when the platform has communities the builder does not personally administer. For MVP, platform admin and community admin are operationally the same person but architecturally distinct roles — this preserves the clean separation for the future without forcing premature governance design.

## 9. Additional Open Questions

These were surfaced during review and merit a thought before implementation, though none block the refactor.

**Legacy route handling.** Old `/threads` and `/thread/{id}` URLs redirect to the corresponding `/c/test/...` routes after migration. No global feed is preserved.

**Public visibility of in-progress signals and votes (RESOLVED).** Full public visibility on public communities. The audit log already exposes everything, partial visibility is security-by-obscurity, and external observers (council members, journalists) need to be able to verify a deliberation independently without joining the community first.

**Vestigial `users.tier` field cleanup.** The migration adds `platform_role` and moves all tier logic to `CommunityMembership.tier`, leaving the original global `users.tier` field in place. After the route migration is complete and verified, a follow-up cleanup migration should drop the dead field. Phase 2 task, not blocking.

## 10. Phase 2 Features (Not Blocking the Refactor)

Features identified as valuable but explicitly deferred until after the community refactor and the first real deliberation cycle.

### Personal Activity Feed

A `/account/activity` view showing recent updates from communities the user belongs to. Chronological order, no engagement-based ranking. Surfaces:
- New threads in the user's communities
- Phase advances on threads the user has signaled or posted in
- New proposals in threads the user has engaged with
- Voting phases opening on threads the user has engaged with
- Facilitator request approvals/denials affecting the user

This is a notifications surface, not a discovery surface. It does not show content from communities the user does not belong to. It does not rank by engagement. It does not aggregate across all communities into a single feed for the platform — it is per-user, scoped to memberships, ordered by time.

The purpose is to let users keep track of deliberations they are actually part of without manually checking each community page. The highest-value updates are phase changes on threads the user has already engaged with, because those signal "you need to come back and participate in the next phase."

### Email-Bridge Contact Feature

The platform does not host direct messages. Instead, a "contact" link on a member's profile page lets another member send one outreach message that is delivered via email. The platform never stores the conversation or hosts the reply — the recipient responds directly to the sender's email address, outside the platform.

Constraints:
- Rate-limited to one outreach per recipient per sender per 30 days
- Sender must be a member of at least one community the recipient is also in
- Recipient can disable contact requests entirely in account settings
- All outreach attempts are logged in the audit log (sender, recipient, timestamp — not message content)

The purpose is to satisfy the legitimate need to reach out to a thoughtful contributor without building the infrastructure (and incurring the moderation, legal, and platform-shape consequences) of a full DM system.

## 11. Potential Future Communities

Communities identified as high-value future deployments. Documented here as candidates, not committed to in the initial migration.

### Bitcoin Quantum Risk
**Type:** Technical
**Verification:** GitHub commit history in relevant repos, vouching by existing members, manual review by community admin
**Founding admin:** TBD — should be a respected Bitcoin developer with existing standing in the community, not the platform builder
**Why it matters:** The Bitcoin developer community has no formal venue for structured deliberation on contested protocol questions. Quantum risk mitigation is shaping up to be the next major contested question, with no clear path to resolution under existing rough-consensus mechanisms. A structured deliberation producing a documented position with explicit dissent would be a referenceable artifact the broader community currently has no way to produce.
**Risks:** Strong personalities and ideological divides; needs a careful verification standard that admits dissenters; outsider-led launch would be received poorly. The platform should provide the infrastructure but the founding admin should come from inside the community.
**Status:** Pitch document to be drafted separately. Not part of initial migration.

### Academic Communities
**Type:** Topical / Institutional
**Verification:** Institutional email + ORCID + admin review of credentials
**Why it matters:** Academic peer review is a weak consensus mechanism with strong status incentives. Structured deliberation could produce a referenceable position artifact that doesn't currently exist in scholarly practice. See the companion document `academic-consensus-mechanisms.md` for a fuller treatment.
**Status:** Speculative. No specific academic community committed to.

### Other Civic Communities
**Type:** Geographic
**Examples:** Other California municipalities, U.S. states with active healthcare or housing policy, counties with active land use disputes
**Status:** Each requires its own founding admin and verification standard. Not committed to until after the Redlands cycle validates the model.

## 12. Recommended Decision Log Entry

```
## 2026-04-11 — Multi-Community Architecture
**Status:** Active
**Domain:** Technical / Strategy
**Context:** The platform's GTM strategy shifted from a national
state-legislature model to a local-first deployment model, starting
with a single municipal community (Redlands, CA). The existing
single-tenant data model cannot support scoped verification, scoped
facilitation, or community-specific deliberation without a refactor.
**Options considered:** (a) Keep single-tenant, add filtering by
location tags; (b) Introduce Community as the primary organizational
unit with per-community tiers, threads, facilitators, and audit logs.
**Decision:** Option (b). Community is the primary unit of organization.
Tiers, threads, facilitator appointments, funding pools, and audit
logs are community-scoped. Domains remain a global taxonomy.
**Reasoning:** The platform's actual nature is multi-tenant deliberation
infrastructure. The state-level theory of change, the local municipal
use case, the HOA use case, and the union use case all require the
same architecture: a bounded group of verified members deliberating
through a phase-gated process. Making Community first-class in the
data model serves all of these without architectural changes.
**Implications:** This is the largest refactor to date. All API routes
must be updated to check community-scoped tiers. URL structure
changes. The community home page becomes the platform's primary
public-facing surface. Existing data is migrated to a default
community with no data loss.
**Revisit if:** The multi-community model creates operational
complexity that a solo builder cannot manage (too many communities
to administer, cross-community coordination needs, etc.).
```
