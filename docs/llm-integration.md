# LLM Integration — Future-Aware Design

## Guiding Principle

The LLM is a **read-only research assistant**, never a deliberation participant.
It summarizes; it does not advocate. It is always visibly labeled.

A participant who never uses the LLM should be able to participate fully.
The LLM should never influence vote outcomes — only help participants
understand the discussion they are part of.

---

## What the LLM May Do (Eventually)

1. **Thread summaries**: "Here are the main points raised in this thread so far"
2. **Proposal analysis**: "Here's how this proposal relates to points raised in discussion"
3. **Signal explanation**: "Participants have expressed these concerns. Here's what they said."
4. **Background corpus Q&A**: "Based on our healthcare corpus, here's relevant context"
5. **Onboarding help**: Explain platform mechanics to new users

## What the LLM Must Never Do

- Post in a thread as if it were a participant
- Recommend a vote choice
- Summarize in a way that amplifies or suppresses specific viewpoints
- Generate proposals
- Have access to individual vote choices or signal attribution

---

## Corpus Structure

The LLM assistant uses two distinct knowledge sources:

### 1. Platform Corpus (Domain Knowledge)
Background documents for each domain (e.g., healthcare policy briefs,
published research). Never includes user-generated content.

```
corpus/
├── healthcare/
│   ├── sources.jsonl          # document metadata
│   ├── chunks/                # chunked text for embedding
│   └── embeddings/            # precomputed vectors
└── housing/
    └── ...
```

Ingestion pipeline (Phase 5):
1. Curate documents manually (editorial control)
2. Chunk with overlap using `langchain.text_splitter`
3. Embed with `text-embedding-3-small` (OpenAI) or local model
4. Store in `pgvector` extension (Supabase supports this)

### 2. Thread Summaries (Generated)
Summaries of user-generated thread content. Generated on demand,
not stored as corpus documents. Clearly labeled as AI-generated.

## Data Architecture for RAG

```sql
-- pgvector extension in Supabase
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE corpus_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    domain_slug TEXT NOT NULL,
    source_title TEXT NOT NULL,
    source_url TEXT,
    chunk_text TEXT NOT NULL,
    embedding vector(1536),  -- OpenAI text-embedding-3-small
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX ON corpus_chunks USING ivfflat (embedding vector_cosine_ops);
```

## API Design for LLM Features

All LLM endpoints are under `/api/v1/assistant/` and are clearly separated
from deliberation endpoints.

```
GET  /api/v1/assistant/thread/{id}/summary
     → Returns AI summary of thread posts
     → Includes disclaimer: "This summary was generated by AI and may omit nuance."
     → Logged to audit_logs with event_type=LLM_SUMMARY_GENERATED

POST /api/v1/assistant/query
     → Body: { domain_slug, question }
     → Returns RAG answer from corpus only (not thread content)
     → Never references individual user posts
```

## Guardrails

### Structural
- LLM responses are never injected into thread post feeds
- All LLM calls logged to audit_logs (event_type=LLM_SUMMARY_GENERATED)
- Summaries are cached for 1 hour (reduce hallucination variance between reads)
- No LLM features available in VOTING phase (zero influence on vote)

### Prompt Engineering
System prompt (non-negotiable, checked in to repo):
```
You are a neutral research assistant for a civic deliberation platform.
Your role is to summarize and explain, never to advocate.
You must not recommend any vote choice or express preferences.
You must not attribute quotes to specific participants.
If you are uncertain, say so.
If a question is outside your corpus, say you don't know.
```

### Human Review
- LLM summaries are flagged if any participant reports them as misleading
- Facilitators can suppress LLM summaries for a thread
- Audit log tracks every LLM call: which user, which thread, timestamp

## Model Selection

| Use case | Model | Rationale |
|---|---|---|
| Thread summaries | `claude-haiku-4-5` | Fast, cheap, sufficient |
| Corpus Q&A (RAG) | `claude-sonnet-4-6` | Better reasoning for nuanced policy questions |
| Embedding | `text-embedding-3-small` | Cost-effective, pgvector compatible |

---

## Integration Sequence (Phase 5)

1. Add `pgvector` to Supabase (one click in dashboard)
2. Create `corpus_chunks` table via Alembic migration
3. Build `scripts/ingest_corpus.py` — reads docs, chunks, embeds, inserts
4. Build `/api/v1/assistant/` routes with RAG retrieval
5. Add assistant UI to web frontend (read-only drawer, clearly labeled)
6. Run end-to-end test: verify no LLM output appears in thread feed
7. Facilitator review: read 10 summaries, flag any that misrepresent discussion
